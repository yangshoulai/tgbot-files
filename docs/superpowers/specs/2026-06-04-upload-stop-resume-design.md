# 上传停止与继续重试设计

日期：2026-06-04
范围：管理后台的本地文件分片上传与 URL 分片上传。

## 背景

当前上传弹窗中，本地文件与 URL 导入都会创建 multipart upload session，然后并发处理分片，最后调用 complete 生成文件索引。现有失败重试机制依赖请求返回错误；如果某个分片请求因远程 URL、Telegram API 或网络链路长时间不返回，前端会一直等待 `Promise.all`，`submitting` 保持为 `true`，弹窗也禁止关闭，用户只能刷新页面，导致本轮已完成分片难以继续利用。

## 目标

1. 本地文件上传和 URL 上传都支持手动停止。
2. 停止后不丢弃已创建的 multipart upload session。
3. 已成功写入后端的分片继续保留；未完成分片可继续重试。
4. 用户无需刷新页面、无需重新选择本地文件、无需重新创建 URL 上传会话即可继续本次上传。
5. 分片请求具备超时边界，避免无人干预时无限挂起。

## 非目标

1. 不尝试删除 Telegram 中已经发送成功的分片消息。
2. 不引入后台任务队列或跨浏览器恢复能力。
3. 不改变 `/f/:token` 的文件访问模型。
4. 不把停止语义持久化为永久取消；停止只是当前前端上传任务的可恢复中断。

## 推荐方案

采用“前端可取消分片队列 + 后端上传状态查询”的组合方案。

前端负责：

- 为每轮上传创建可取消上下文。
- 为每个分片请求创建 `AbortController`。
- 点击停止后 abort 正在等待的分片请求，并停止派发新分片。
- 把已完成分片、失败分片、未派发分片合并为 retry 状态。
- 重试时复用原 `uploadId`，只处理后端确认缺失的分片。

后端负责：

- 提供 upload session 状态查询接口，返回已上传分片和缺失分片。
- 保持现有 chunk upsert 与 complete 语义不变。

## API 设计

新增管理端接口：

```http
GET /api/admin/uploads/:id/status
```

响应示例：

```json
{
  "ok": true,
  "upload": {
    "id": "...",
    "source_kind": "url",
    "file_name": "video.mp4",
    "mime_type": "video/mp4",
    "size": 104857600,
    "chunk_size": 10485760,
    "chunk_count": 10,
    "direct_access": true,
    "directory_path": "/"
  },
  "uploaded_chunks": [0, 1, 2],
  "missing_chunks": [3, 4, 5, 6, 7, 8, 9]
}
```

状态查询只读取数据库，不触发远程 URL 或 Telegram 请求。`source_kind` 用于确保本地上传与 URL 上传不会误用彼此的 chunk 接口。

## 前端数据流

### 初始化上传

1. 本地文件：调用 `initMultipartUpload()`。
2. URL：调用 `initUrlMultipartUpload()`。
3. 根据返回的 upload 信息创建分片状态列表。
4. 创建当前上传任务上下文。

### 分片处理

1. 队列从待上传分片中取 index。
2. 每个分片请求创建独立 `AbortController`。
3. 请求同时挂载超时控制。
4. 成功后标记 completed。
5. 失败、超时或 abort 后标记 failed。
6. 如果用户已经停止，不继续派发新分片。

### 手动停止

1. 设置当前任务 `cancelled=true`。
2. abort 所有 active controllers。
3. 当前运行分片会进入 failed 或 cancelled 结果。
4. 队列结束后调用状态查询接口同步后端真实分片列表。
5. UI 标记为 error，并显示“已停止，可重试未完成分片”。
6. retry 状态记录 `uploadId`、`chunkSize`、`chunkCount`、后端已完成分片、缺失分片。

### 继续重试

1. 点击“重试失败分片”。
2. 先调用状态查询接口同步真实完成状态。
3. 只对缺失分片继续执行分片队列。
4. 全部分片完成后调用 `completeMultipartUpload()`。

## 超时策略

- 本地分片请求默认 5 分钟超时。
- URL 分片请求默认 10 分钟超时。
- 超时按可重试失败处理，不直接放弃 upload session。

## UI 设计

本地文件行与 URL 上传行都增加上传中停止按钮：

- 本地上传中：`停止上传`
- URL 导入中：`停止导入`
- 停止后：展示错误/提示文案 `已停止，可重试未完成分片`
- 继续使用现有 `重试失败分片` 按钮补传缺失分片

弹窗底部在上传中保留一个全局停止按钮，用于停止当前正在执行的上传任务。停止完成后允许关闭弹窗；只要页面不刷新，仍可继续重试。

## 错误处理

- `AbortError` 且任务处于 cancelled 状态：显示为用户停止，不作为危险错误 toast。
- 非用户停止的网络错误、5xx、429：保留现有可重试逻辑。
- 状态查询失败：仍使用前端已知 completed/failed 状态构建 retry，但提示用户状态同步失败。
- complete 前再次检查后端状态，避免因前端 abort 后服务端实际成功写入造成误判。

## 兼容性与风险

- 浏览器 abort 只能保证前端不再等待；如果服务端已经开始向 Telegram 上传，该后端请求可能自然完成。因此必须通过状态查询接口对齐数据库真实状态。
- 重试缺失分片可能导致某些已成功但未及时写入数据库的分片被重复发送到 Telegram。该风险目前已经存在，状态查询能降低但不能彻底消除极端竞态。
- 不删除旧分片消息，符合当前“软删除/不动 Telegram 原消息”的项目语义。

## 测试计划

1. 前端类型检查：`pnpm typecheck`。
2. 前端构建：`pnpm build:frontend`。
3. 后端测试：补充 status 接口测试，执行 `pnpm test`。
4. 手动验证：
   - 本地大文件上传中停止，随后重试完成。
   - URL 上传中停止，随后重试完成。
   - 停止后关闭弹窗不会导致页面锁死。
   - 超时或 abort 后分片进入可重试状态。
